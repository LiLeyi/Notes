    并发控制

    介绍

    多事务执行方式：

    • 串行：每一个时刻只能有一个事务运行。不能充分利用资源
    • 交叉办法：并行事务的并行操作轮流交叉运行，是但单处理机系统中的并发方式，能够减少处理机的空闲时间，提高系统效率
    • 同时并发方式：多机处理系统，每个处理机可以运行一个事务，多个处理机可以同时运行多个事务，实现多个事务真正的并行

    并发带来的问题：

    • 可能会存取和存储不正确的数据，破坏事务的隔离性和数据库一致性。DBMS必须提供并发控制机制

    问题：

    • 丢失更新：例如两进程分别修改某一个值，例如减一，可能最后结果仍然是减一
    • 不可重复读：事务T1读取数据后，T2对同一数据执行更新操作，使T1再次读取该数据时，得到前一次不同的值。或者少、多了记录条数（称为幻影现象）
    • 读脏数据：事务1修改某一数据并写回磁盘。事务2读取同一数据，事务1由于某原因被撤销，则事务2得到不正确数据

    原因：并发操作破坏了事物的隔离性，导致破坏了一致性。并发控制机制要用正确的方式调度并发操作，使一个用户事务的执行不受其他事务的干扰

    计算机系统对并行事务中并行操作的调度是随机的，而不同的调度可能会导致产生不同结果。但以不同顺序串行执行事务也可能会产生不同的结果，但由于不会将数据库置于不一致状态，所以都可以认为是正确的。

    多个事务的并发执行是正确的，当且仅当并发执行的结果与这些事务按某一串行顺序执行的结果相同，这种调度策略被称为可串行化调度。

    可串行性是并行事务正确性的唯一准则，即可串行化等价于正确性。所以DBMS要提供策略使得可串行化

    调度的冲突等价性：

    冲突操作是指不同的事务对同一个数据的读写操作和写写操作。其他操作不冲突。不同事务的冲突操作和同一事物的两个操作不能交换

    如果一个调度S能经过一系列非冲突操作执行顺序的交换变成调度S1，则称调度S和S1冲突等价

    例如：

    \[S = R_1(A) W_1(A) R_2(A) W_2(A) R_1(B) W_1(B) R_2(B) W_2(B)\]

    把\(W_2(A),R_2(A)\)与\(R_1(B)W_1(B)\)依次调换可得串行操作，即S是可串行化的，L和S是冲突等价的

    可串行化的充分条件：一个调度S在保证冲突操作的次序不变的情况下，通过交换两个事务不冲突的操作的次序得到另一个调度S'，如果S'是串行的，称调度S为冲突可串行化的调度。但有不满足冲突可串行化条件的调度，是可串行化

    解决

    概述

    • 封锁：一个事务存取某个数据项时，不允许其他事务存取。
    • 时间戳
    • 乐观控制法

    封锁

    锁分类：

    • 共享锁（S锁，写锁）：若已有S锁，其他事务只能再加S锁而不能加X，直到该锁释放S锁
    • 排它锁（X锁，读锁）：若T对数据对象加X锁，则只允许T读写A，而其他无法加任何锁
    \(T_2 / T_1\) X S -
    X N N Y
    S N Y Y
    - Y Y Y

    封锁协议概念：运用封锁方法时，对数据对象加锁时需要约定一些规则

    • 何时申请
    • 持续时间
    • 何时释放

    封锁协议

    • 一级封锁协议：事务T修改A之前必须先加X锁，事务结束再释放。问题：不能保证可重复读、不读脏数据（因为不上锁时能读X锁住的内容）
    • 二级封锁协议：事务T在读数据A之前必须先对其加S锁，读完释放。同时也要满足一级封锁协议。可以防止读脏数据
    • 三级封锁协议：在二级封锁协议基础上，某一事务施加的S锁要保持到该事务结束时才释放

    协议越高，并发性能越差

    活锁：在数据库系统中，某个事物由于请求封锁，但总得不到锁而处于长时间等待。解决：采用先来先服务，按请求封锁的先后次序对这些事务排队

    死锁：处于等待状态的两个或以上事务相互封锁对方请求的资源，使得没有任何一个事务可以获取足够的资源运行完毕。解决方法：

    • 预防死锁：操作系统中适用但数据库中不适用
    • 一次封锁法：将事务在执行过程中可能要封锁的数据对象全部加锁。会降低并发度，而且很难事先确定每个事物所要封锁的数据对象
    • 顺序封锁法：预先对数据对象规定封锁顺序，所有事务都要按这个顺序实行封锁。维护成本高、难以实现（因为很难按规定顺序去加锁，有时会动态加锁）
    • 事务重试法：当事务T2申请的锁已经被T1占有时，根据事务开始的先后，将T1释放的锁授予T2，然后T1回滚，自动重试。
    • 允许发生，诊断与接触：更常用

    定期检测系统中是否存在死锁。检测到有死锁，就要解除。选择处理代价最小的死锁，回滚，然后让其他事务运行，再重新执行

    • 超时法：如果一个事务等待时间超过了规定的实现，就认为发生了死锁。实现简单，但可能误判死锁，时间设置太长死锁不能及时发现
    • 事务等待图法。有向图，V表示正在运行的事务，U表示事务等待的情况。若V1等待V2，则画一条有向边，从V1指向V2。并发子系统周期性地（例如1min）检测事务等待图，如果发现图中存在回路，则表示系统中出现死锁。检测算法：拓扑排序。仍然通过回滚一个或多个相关事务实现

    两阶段封锁协议2PL：最常用。理论上可证明产生的是可串行化调度。事务必须分两个阶段对数据项加锁、解锁

    • 在对任何数据进行读写之前，事务首先要获得对该数据的封锁
    • 在释放一个封锁之后，事务不再获得任何其他封锁

    两个阶段：

    • 扩展阶段：获得封锁。此阶段不能释放任何锁
    • 收缩阶段：释放封锁。此阶段不能再申请任何锁

    事务遵循两段封锁协议是可串行化调度的充分条件，而不是必要

    \[\text{2PL} \Rightarrow \text{冲突等价} \Rightarrow \text{可串行化} \Leftrightarrow \text{正确性}\]

    两阶段封锁协议不能防止死锁（一次封锁法满足两阶段封锁协议，但不是所有两阶段封锁协议都是一次封锁法）

    一次封锁法也可以算一种协议

    锁的请求、授予、解除由数据库系统的锁管理器完成。维护一个记录链表，按请求先后顺序排列每个封锁请求

    多粒度封锁

    封锁对象的大小称为封锁粒度。封锁对象：

    • 属性值
    • 属性值集合
    • 元组
    • 关系
    • 索引项
    • 整个数据库
    • ...

    多粒度封锁：

    • 封锁粒度越大
    • 封锁的对象越少
    • 并发度越小
    • 系统开销越小

    多粒度树：

    • 以树形结构表示多级封锁粒度
    • 根节点是整个数据库，表示最大数据粒度
    • 叶结点表示最小数据粒度

    多粒度封锁协议：允许多粒度树中每个节点独立加锁（意味着其后裔节点也同时加锁）。两种封锁：

    • 显式封锁：直接加到数据节点的锁
    • 隐式封锁：由于上级节点加锁而使该数据对象上了锁

    缺点：需要向上查找是否加锁。检测加锁顺序：

    • 锁表检查显式封锁
    • 检查该锁表是否与上层的显示封锁冲突
    • 检查该锁表是否与下层的显示封锁冲突

    为了增加效率，引入意向锁：对任意节点加基本锁，必须先对其上层节点加意向锁。如果对一个节点加意向锁，说明该节点的下层节点正在被加锁。例如对元组加锁时，先给所在关系加意向锁，避免了向下层搜索（因为越下层对象数量越多）。常见意向锁

    • 意向共享锁，IS锁：如果对一个数据对象加IS锁，表示其后裔节点拟加S锁。例如对某个元组加S锁，则需要对关系、数据库加IS锁
    • 意向排他锁，IX锁：如果对一个数据对象加IX锁，表示其后裔节点拟加X锁
    • 共享意向排他锁，SIX锁：如果对一个数据对象加SIX锁，表示对这个数据对象加S锁，再加IX锁。例如对一个表加SIX锁，则表示该事务要读整个表，且某些元组会进行修改。相容矩阵：
    T1/T2 S X IS IX SIX -
    S Y N Y N N Y
    X N N N N N Y
    IS Y N Y Y Y Y
    IX N N Y Y N Y
    SIX N N Y N N Y
    - Y Y Y Y Y Y

    锁的强度：X>SIX>S>IS、X>SIX>IX>IS。强度指排斥过程。一个事务在申请封锁时强锁代替弱锁是安全的。IS和IX锁是相融的，因为下级会再次检查基本锁是否冲突

    申请封锁应该自上而下进行，释放时应该自下而上

    提高了并发度，减少了加锁、解锁的开销
