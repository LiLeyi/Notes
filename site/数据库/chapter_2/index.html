    数据模型

    E-R概念模型

    概念模型用途：

    • 信息世界的建模
    • 现实世界到机器世界的一个中间层次
    • 是设计数据库的有力工具
    • 是数据库设计人员和用户之间进行交流的语言

    对概念模型的基本要求：

    • 较强的语义表达能力，能够方便、直接地表达应用中的各种语义知识
    • 简单、清晰、易于用户理解

    实体：客观存在并可相互区别的事物称为实体

    属性：实体所具有的某一特征称为属性

    联系：在现实世界中，事物内部以及事物之间是有联系的。在信息世界中将被抽象为实体内部的联系和实体之间的联系。分类：

    • 实体内部的联系通常是指组成实体的各属性之间的联系；
    • 实体之间的联系通常是指不同实体集之间的联系
    • 1:1联系
    • 1:n联系
    • m:n联系

    注意：多个实体的多对多关系和多个实体两两多对多不同

    graph TD
    A[实体A] --> ABC[三元关系ABC]
    B[实体B] --> ABC
    C[实体C] --> ABC

    两两多对多：

    graph LR
    A <--> B
    B <--> C
    A <--> C

    同一实体集内部也有一对一、一对多、多对多的联系

    E-R图：

    • 矩形：实体，可以有属性
    • 弱实体：这种实体的存在依赖其他实体的存在，依赖的实体称为强实体。弱实体用双框矩形，且与关系的链接用双线（表示全部参与，因为弱实体依赖于强实体）
    • 子类实体：矩形框两竖边为双线，与超类的线上加小圈（中间不需要菱形）
    • 椭圆：属性
    • 菱形：联系，也可以存在属性。
    • 基数比约束（1：1，,1：n，m：n）
    • 参与约束：根据实体集中的实体是否全部参与联系来描述实体参与联系的约束。一个实体集中所有实体都参与联系称为完全参与，否则称为部分参与
    • 参与度：实体参与联系的最小、最大次数，称为实体参与度，记为(min,max)

    层次数据模型

    典型：IBM公司的IMS数据库管理系统

    用树型模型结构表示各类实体以及实体间的联系

    条件：

    • 只有一个根节点
    • 其他每个节点都有父节点

    优点：

    • 简单清晰
    • 查询效率高
    • 提供良好的数据完整性支持

    缺点：

    • 不能直接表示两个以上实体间复杂联系、实体间的多对多关系
    • 对数据的插入、删除限制多
    • 查询子女节点必须通过双亲节点
    • 由于结构严密，层次命令趋于程序化

    网状数据模型

    • 允许一个以上的节点无双亲
    • 一个节点可以有多余一个的双亲

    优点：

    • 能更好描述现实世界（层次模型是网状模型的特例）
    • 有良好的效率，存取效率高

    缺点：

    • 结构比较复杂，随着应用环境扩大，数据库的结构会变得越来越复杂，不利于最终用户掌握
    • DDL、DML语言（Data Definition Language、Data Manipulation Language）复杂，用户不容易使用，访问数据时必须自行选择存取路径，增加了程序编写的负担

    关系模型

    IBM的Edgar Frank Codd在1970年提出的

    基本数据结构被限制为二维表（有多个行列组成），一张二维表称为一个关系。

    一个关系可以描述一个实体，联系也可以为二维表

    • 属性：一个关系有多个列（每个列之间无重复，无序），每一列为关系的一个属性
    • 域：属性的取值范围
    • 元组：关系是元组的集合（也就是无序、不重复）。一个元组对应实体集中的一个个体。一个元组由若干个分量组成，一个分量对应一个属性值
    • 键：是一个或多个属性组成，能够唯一标识一个元组。一个关系中可能有多组属性能够起到标识元组的作用，所以一个关系可能有多个键。最重要的键称为主键
    • 关系模式：对关系结构的描述。用如下形式：关系名（属性名1，属性名2，...，属性名m）

    关系数据库模式是一组关系模式的集合。这组关系模式对应的关系的集合称为关系数据库

    概念模式与逻辑模式的对应

    • 实体集：关系表
    • 实体集的属性：表的列
    • 联系集+属性：表和列
    • 实体的实例：表的各行
    • 键：能唯一标识元组的属性或属性组
    • 不含多余属性的超键称为候选键
    • 如果一个关系中有多个候选键，则可选一个作为主键
    • 若关系的键由多个属性组成，称为联合键
    • 关系的所有属性构成该关系的键，称为全键
    • 数据操作：SQL语句、关系代数、关系演算
    • 联系类型约束（联系类型约束）：实体完整性
    • 键约束：参照完整性
    • 完全参与约束：用户自定义完整性

    补充：

    • 规范：关系的每一个分量都要是不可分割的数据项。（一范式）
    • 完整性约束：实体完整性、参照完整性、用户自定义完整性
    • 关系模型的数据操纵：对关系中的数据可进行查询、插入、删除、修改。对数据库的全部操作都可以归结为对关系的运算

    优点：

    • 结构简单
    • 一体化数据子语言
    • 数据独立性高
    • 面向集合的存取方式
    • 坚实的理论基础
    • 有利于开展其他应用
